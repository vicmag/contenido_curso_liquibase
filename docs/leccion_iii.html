DDL VS DML: ¿CUÁL ES LA DIFERENCIA?

DDL

Data Definition Language

Define la estructura

Comandos que definen la estructura de la base de datos:

CREATE
Crear objetos (tablas, vistas, índices)
ALTER
Modificar estructura de objetos existentes
DROP
Eliminar objetos de la base de datos
TRUNCATE
Vaciar una tabla manteniendo la estructura
RENAME
Renombrar objetos

Características: Suelen requerir menos frecuencia de ejecución, tienen mayor impacto, generalmente requieren permisos elevados.

DML

Data Manipulation Language

Manipula los datos

Comandos que manipulan los datos dentro de las tablas:

INSERT
Agregar nuevos registros a una tabla
UPDATE
Modificar registros existentes
DELETE
Eliminar registros de una tabla
SELECT
Consultar datos (solo lectura)
MERGE
Insertar o actualizar según condición

Características: Se ejecutan con más frecuencia, tienen menor impacto estructural, pueden ser reversibles.

Aspecto DDL DML
Propósito Definir estructura Manipular datos
Frecuencia en changesets Menos frecuente Más frecuente
Reversibilidad Difícil (DROP) Posible (DELETE/UPDATE)
Impacto en Liquibase Cambia DATABASECHANGELOG No afecta tracking
Ejemplo común CREATE TABLE usuarios INSERT INTO usuarios VALUES(...)

EJEMPLOS PRÁCTICOS PARA CHANGESETS

-- liquibase formatted sql
-- changeset autor:maria.gomez id:001-1
-- comment: Crear tabla de productos para catálogo

CREATE TABLE IF NOT EXISTS productos (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  codigo VARCHAR(20) NOT NULL UNIQUE,
  nombre VARCHAR(100) NOT NULL,
  descripcion TEXT,
  precio DECIMAL(10,2) NOT NULL,
  stock INTEGER DEFAULT 0,
  activo BOOLEAN DEFAULT true,
  creado_en TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  actualizado_en TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- rollback DROP TABLE IF EXISTS productos;
Puntos clave:
1. IF NOT EXISTS hace el script idempotente
2. UUID con gen_random_uuid() es buena práctica en YugabyteDB
3. Se especifican constraints (NOT NULL, UNIQUE)
4. Incluye campos de auditoría (creado_en, actualizado_en)
5. Rollback explícito definido en comentario
-- changeset autor:maria.gomez id:002-1
-- comment: Agregar columna categoría a productos

ALTER TABLE productos
  ADD COLUMN categoria_id UUID;

-- rollback ALTER TABLE productos DROP COLUMN categoria_id;

-- changeset autor:maria.gomez id:002-2
ALTER TABLE productos
  ADD CONSTRAINT fk_producto_categoria
  FOREIGN KEY (categoria_id)
  REFERENCES categorias(id);

-- rollback ALTER TABLE productos DROP CONSTRAINT fk_producto_categoria;
Puntos clave:
1. Se separan cambios en diferentes changesets para granularidad
2. Cada ALTER tiene su propio rollback definido
3. Se agrega columna primero, luego la foreign key
4. Nombrado consistente de constraints (fk_tabla_columna)
5. Orden lógico: estructura antes de relaciones
-- changeset autor:carlos.lopez id:003-1
-- comment: Insertar categorías iniciales del sistema

INSERT INTO categorias (id, nombre, descripcion) VALUES
  ('a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11', 'Electrónica', 'Dispositivos electrónicos'),
  ('b1eebc99-9c0b-4ef8-bb6d-6bb9bd380a12', 'Ropa', 'Prendas de vestir'),
  ('c2eebc99-9c0b-4ef8-bb6d-6bb9bd380a13', 'Hogar', 'Artículos para el hogar')
ON CONFLICT (id) DO NOTHING;

-- rollback DELETE FROM categorias WHERE id IN (
-- 'a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11',
-- 'b1eebc99-9c0b-4ef8-bb6d-6bb9bd380a12',
-- 'c2eebc99-9c0b-4ef8-bb6d-6bb9bd380a13'
-- );
Puntos clave:
1. ON CONFLICT DO NOTHING hace el INSERT idempotente
2. Se usan UUIDs explícitos para consistencia entre ambientes
3. Inserción múltiple en una sola sentencia para eficiencia
4. Rollback explícito aunque sea DML (buena práctica)
5. Ideal para datos maestros/configuración inicial
-- changeset autor:ana.martinez id:004-1
-- comment: Actualizar precios por ajuste de inflación

UPDATE productos
SET precio = precio * 1.05,
    actualizado_en = CURRENT_TIMESTAMP
WHERE activo = true;

-- rollback UPDATE productos SET precio = precio / 1.05
-- WHERE activo = true AND actualizado_en >= '2024-01-01';
Puntos clave:
1. WHERE específico para evitar actualizar todos los registros
2. Se actualiza también el campo de auditoría (actualizado_en)
3. Rollback matemáticamente reversible
4. Condición en rollback más específica para seguridad
5. DML condicional es común en migraciones de datos

CONSIDERACIONES ESPECIALES PARA YUGABYTEDB

gen_random_uuid()
Función nativa para UUIDs en YugabyteDB
ON CONFLICT
Manejo de conflictos en INSERTs distribuidos
HASH/ASC/DESC
Especificar ordenamiento para distribuir datos
SPLIT AT
Particionamiento explícito de tablas
-- Ejemplo YugabyteDB: Creación de tabla con particionamiento
CREATE TABLE ventas (
  id UUID DEFAULT gen_random_uuid(),
  fecha_venta DATE NOT NULL,
  monto DECIMAL(12,2),
  PRIMARY KEY (id HASH, fecha_venta ASC)
) SPLIT AT VALUES (
  ('2024-01-01'),
  ('2024-07-01')
);

EJECUCIÓN EN DIFERENTES AMBIENTES

1

Ambiente de Desarrollo

DEV

Propósito: Desarrollo y pruebas iniciales

Características: Datos sintéticos, reset frecuente, debug habilitado

Ejecución: Manual desde IDE o línea de comandos

Responsable: Desarrollador

2

Ambiente de Testing

TEST QA

Propósito: Pruebas automatizadas y QA

Características: Datos representativos, versiones estables

Ejecución: Automática por pipeline de integración

Responsable: Pipeline CI/CD

3

Ambiente de Producción

PROD

Propósito: Sistema en vivo con usuarios reales

Características: Datos reales, alta disponibilidad, backups

Ejecución: Automática con aprobación manual

Responsable: DevOps con validación

Diferencias en la Ejecución

# Comando Liquibase para desarrollo (local)
liquibase \
  --changeLogFile=changelog_master.json \
  --url="jdbc:postgresql://localhost:5433/dev_db" \
  --username=dev_user \
  --password=dev_pass \
  update

# Comando Liquibase para producción (pipeline)
liquibase \
  --changeLogFile=changelog_master.json \
  --url="${PROD_DB_URL}" \
  --username="${PROD_DB_USER}" \
  --password="${PROD_DB_PASS}" \
  --defaultsFile=properties/prod-instance.json \
  update

BUENAS PRÁCTICAS PARA SQL EN CHANGESETS

✔ IDEMPOTENCIA
Usar IF NOT EXISTS, ON CONFLICT DO NOTHING
✔ ROLLBACK
Siempre incluir comentario de rollback
✔ GRANULARIDAD
Un changeset por cambio lógico
✔ VALIDACIÓN
Probar localmente antes de commit
⚠ EVITAR DROP
Preferir ALTER sobre DROP TABLE
⚠ TRANSACCIONES
Cada changeset es una transacción

ACTIVIDAD PRÁCTICA

Ejercicio: Identifica qué tipo de comando SQL (DDL/DML) es cada uno y propón un ejemplo para Liquibase:

  1. Agregar una columna "telefono" a la tabla "clientes"
  2. Insertar 3 roles iniciales del sistema: admin, usuario, invitado
  3. Crear un índice en la columna "email" de la tabla "usuarios"
  4. Actualizar el estado de pedidos antiguos a "completado"
  5. Crear una tabla "log_auditoria" para tracking
Solución:
1. DDL - ALTER TABLE clientes ADD COLUMN telefono VARCHAR(15);
2. DML - INSERT INTO roles (id, nombre) VALUES (...) ON CONFLICT DO NOTHING;
3. DDL - CREATE INDEX idx_usuarios_email ON usuarios(email);
4. DML - UPDATE pedidos SET estado='completado' WHERE fecha < '2024-01-01';
5. DDL - CREATE TABLE log_auditoria (...);

Recordatorio: Cada uno sería un changeset separado en Liquibase.
