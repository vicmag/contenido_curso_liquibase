PIPELINES EN AZURE DEVOPS

Automatización de despliegues de base de datos con Liquibase

OBJETIVOS DE APRENDIZAJE

  • Comprender la arquitectura de pipelines en Azure DevOps para Liquibase
  • Configurar triggers automáticos basados en cambios en GitHub
  • Gestionar variables de entorno y secretos por ambiente
  • Implementar aprobaciones manuales para producción
  • Monitorear y resolver fallos en la ejecución de pipelines
  • Configurar notificaciones y alertas automáticas

ARQUITECTURA DEL PIPELINE DE DESPLIEGUE

1

Source Control (GitHub)

Repositorio con changelogs y changesets

Trigger en push a main/master
Validación de estructura de archivos
Checkout automático del código
2

Build & Validate

Validación y preparación de cambios

Instalación de Liquibase
Validación sintáctica de JSON/SQL
Pruebas de consistencia
Generación de reportes
3

Deploy to Test

Despliegue automático a ambiente de testing

Conexión a YugabyteDB Test
Ejecución de changesets pendientes
Validación de cambios aplicados
Ejecución de pruebas automatizadas
4

Approval Gate

Aprobación manual para producción

Notificación a responsables
Revisión de cambios aplicados en test
Aprobación/rechazo manual
5

Deploy to Production

Despliegue final a ambiente productivo

Conexión a YugabyteDB Prod
Ejecución validada de changesets
Verificación final
Notificación de completado
Automático
Requiere aprobación
Rollback automático en fallo

CONFIGURACIÓN YAML DEL PIPELINE

# azure-pipelines.yml
# Pipeline para despliegue de cambios en base de datos

trigger:
  branches:
    include:
      - main
      - releases/*
  paths:
    include:
      - scripts/**
      - changelog_master.json
      - properties/**

variables:
  - group: 'liquibase-global-vars'
  - name: liquibaseVersion
    value: '4.20.0'

stages:
  - stage: Validate
    displayName: 'Validar cambios'
    jobs:
      - job: ValidationJob
        pool:
          vmImage: 'ubuntu-latest'
        steps:
          - task: UsePythonVersion@0
            inputs:
              versionSpec: '3.9'
          - script: |
                # Validar estructura JSON
                python validate_changelogs.py
            displayName: 'Validar changelogs'
- stage: DeployToTest
  displayName: 'Desplegar en Test'
  dependsOn: Validate
  condition: succeeded()
  jobs:
    - job: DeployTestJob
      pool:
        vmImage: 'ubuntu-latest'
      steps:
        - task: DownloadSecureFile@1
          name: liquibaseJar
          inputs:
            secureFile: 'liquibase.jar'
        - script: |
            java -jar $(liquibaseJar.secureFilePath) \
              --defaultsFile=properties/test-instance.json \
              --changeLogFile=changelog_master.json \
              update
          displayName: 'Ejecutar Liquibase en Test'

- stage: DeployToProd
  displayName: 'Desplegar en Producción'
  dependsOn: DeployToTest
  condition: succeeded()
  jobs:
    - deployment: DeployProd
      environment: 'yugabyte-prod'
      strategy:
        runOnce:
          deploy:
            steps:
              - script: |
                  java -jar $(liquibaseJar.secureFilePath) \
                  --defaultsFile=properties/prod-instance.json \
                  --changeLogFile=changelog_master.json \
                  update

TRIGGERS Y EVENTOS

Push a Main

Cualquier commit en main/master

Nuevo Tag

Tags semánticos v1.0.0

Programado

Ejecución diaria/noche

Manual

Inicio manual por usuario

Configuración de Path Filters:
El pipeline solo se activa cuando hay cambios en:
  • changelog_master.json - Cambios en estructura
  • scripts/** - Nuevos changesets SQL
  • properties/** - Cambios en configuración

GESTIÓN DE VARIABLES Y SECRETOS

Variables de Grupo

liquibase-global-vars

Configuración compartida entre pipelines:

  • liquibaseVersion: "4.20.0"
  • jdbcDriver: "org.postgresql.Driver"
  • logLevel: "INFO"

Secretos por Ambiente

DEV TEST STAGING PROD

Credenciales almacenadas en Azure Key Vault:

$(DB_PASSWORD)
$(API_KEY)
$(SSL_CERT)

Variables de Pipeline

$(Build.SourceVersion)

Commit ID que disparó el pipeline

$(Build.BuildId)

ID único de ejecución

$(System.StageName)

Nombre del stage en ejecución

SEGURIDAD

Nunca almacenes credenciales en el código. Usa siempre:

  • Azure Key Vault para secretos
  • Variable Groups en Library
  • Secure Files para certificados
  • Service Connections para autenticación

APROBACIONES Y GATES

Type de Gate Propósito Configuración Ejemplo
Pre-deployment Approval Aprobación manual antes del despliegue Usuario/Grupo específico DBA Senior debe aprobar
Post-deployment Approval Validación después del despliegue Automático o manual Verificar cambios aplicados
Function Gate Validación funcional automática Script/API validation Probar conexión a DB
Schedule Gate Restricción por horario Ventana de despliegue Solo entre 22:00-06:00
Configuración de Aprobaciones en Producción:
  1. Crear Environment "yugabyte-prod" en Azure DevOps
  2. Agregar approvers: DBA Team, DevOps Lead
  3. Configurar timeout: 4 horas (para evitar bloqueos)
  4. Habilitar notificaciones por email y Teams
  5. Requerir al menos 2 aprobaciones

MONITOREO Y RESOLUCIÓN DE PROBLEMAS

Errores Comunes

Connection Failure

DB no disponible o credenciales inválidas

SQL Syntax Error

Error en changeset SQL

Validation Warnings

Problemas en changelog JSON

Herramientas de Debug

  • Pipeline Logs: Descarga completa de logs
  • Liquibase Log: Nivel DEBUG para troubleshooting
  • YugabyteDB Logs: Consultar en plataforma
  • Validation Scripts: Pre-flight checks
  • Dry Run: updateSQL para previsualizar

Alertas y Notificaciones

  • Email: Equipo de base de datos
  • Microsoft Teams: Canal #db-deployments
  • Slack: Canal de alertas
  • Azure Monitor: Métricas y alertas
  • Webhooks: Sistemas externos

ACTIVIDAD PRÁCTICA

Escenario: Necesitas configurar un pipeline para un nuevo proyecto. Identifica qué configuraciones necesitas:

  1. ¿Qué triggers configurarías para desarrollo vs producción?
  2. ¿Qué variables necesitarías almacenar en Key Vault?
  3. ¿Qué aprobaciones configurarías para producción?
  4. ¿Cómo manejarías rollbacks automáticos?
  5. ¿Qué notificaciones configurarías para el equipo?
Solución Sugerida:

1. Triggers: Push a main para DEV/TEST, solo tags para PROD
2. Key Vault: DB passwords, API keys, SSL certificates
3. Aprobaciones: DBA Lead + Application Owner
4. Rollbacks: Configurar stage de rollback automático en fallo
5. Notificaciones: Teams para éxito/error, email para aprobaciones pendientes

Configuración adicional: Ventana de mantenimiento de 22:00-06:00 para PROD
